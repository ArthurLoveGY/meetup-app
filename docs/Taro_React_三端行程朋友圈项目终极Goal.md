# Taro + React 三端「朋友圈式行程协作」项目终极 Goal 文档

> 项目代号：TripCircle / 朋友行程圈  
> 技术方向：Taro + React + TypeScript 作为统一前端体系，优先覆盖 Android、iOS、微信小程序。  
> 目标平台：Android App、iOS App、微信小程序  
> 产品定位：一个围绕“熟人小圈子”的行程发布、朋友参加、约定协作和活动记录平台。

---

## 1. 项目愿景

打造一个属于用户自己的“小型朋友圈 + 行程协作空间”。

用户可以创建一场有明确结构、有时间线、有参与规则的行程，例如周末自驾、城市散步、聚餐、旅行、露营、看展、运动、游戏开黑、短途出游等。朋友看到后，可以选择想去、确认参加、暂不确定、评论、提出建议、确认时间、同步状态，最终形成一次清晰、轻量、好约、不混乱的活动组织流程。

这个项目不是普通日历，也不是单纯朋友圈，而是介于以下几类产品之间：

- **朋友圈**：有熟人关系、有动态、有互动。
- **日历 / 行程工具**：有时间、地点、提醒、安排。
- **活动报名工具**：有参与状态、人数限制、确认机制。
- **旅行计划工具**：有路线、预算、清单、准备事项。
- **私域社交工具**：用户可以维护自己的朋友分组和关系圈。

最终目标是让用户不再通过微信群里零散聊天来约行程，而是通过一个结构化、可视化、可追踪的方式，把“想约朋友一起做某件事”变得简单、清楚、有仪式感。

---

## 2. 核心一句话

> 让用户可以像发朋友圈一样发布一场有计划的行程，让朋友像参加活动一样确认、讨论、协作和赴约。

## 3. 当前平台范围

本阶段只考虑以下三个平台：

| 平台 | 目标形态 | 推荐实现路线 | 优先级 | 说明 |
|---|---|---|---|---|
| 微信小程序 | 微信生态内的轻量入口 | Taro 编译到微信小程序 | P0 | 最适合作为 MVP 首发平台，便于好友分享、扫码加入、微信登录 |
| Android | 原生移动 App | Taro + React Native | P1 | 面向长期 App 化，适合推送、系统通知、原生体验 |
| iOS | 原生移动 App | Taro + React Native | P1 | 与 Android 保持一致的移动端能力，后续需考虑 Apple 登录与审核规则 |

暂不考虑：

- macOS 桌面端。
- Windows 桌面端。
- 独立 Web H5 正式产品形态。
- 支付宝 / 抖音 / 百度等其他小程序平台。

Web H5 可以作为内部调试或管理后台的技术储备，但不进入当前核心目标。

## 4. 技术总原则

### 4.1 推荐技术路线

```txt
同一套核心业务代码
        │
        ├── 微信小程序：Taro + React + TypeScript
        │
        ├── Android：Taro + React Native
        │
        └── iOS：Taro + React Native
```

### 4.2 项目技术栈建议

| 层级 | 推荐方案 | 说明 |
|---|---|---|
| 跨端框架 | Taro | 统一管理小程序与 React Native 端代码 |
| UI 框架 | NutUI React / Taroify / 自研组件 | 优先选择适配 Taro 的组件库，核心组件建议逐步自研 |
| 语言 | TypeScript | 所有业务模块强类型化 |
| 前端框架 | React | 使用函数组件 + Hooks |
| 状态管理 | Zustand / Jotai / Redux Toolkit | 推荐 Zustand，轻量且适合中小型项目 |
| 网络请求 | Taro.request 封装 / Axios 适配层 | 统一 request、token、错误处理、重试机制 |
| 表单 | React Hook Form / 自研轻量表单 | 行程编辑器会大量使用表单能力 |
| 日期时间 | dayjs | 处理日期、时间、倒计时、格式化 |
| 地图位置 | 腾讯位置服务 / 微信小程序地图能力 | 小程序端优先微信生态，App 端后续适配原生地图 SDK |
| 图片上传 | 对象存储 OSS / COS / S3 | 头像、封面图、相册图片 |
| 后端接口 | RESTful / tRPC / GraphQL 任选 | 初期推荐 RESTful，便于调试与移动端适配 |
| 后端服务 | Node.js NestJS / Java Spring Boot / Go | 推荐 NestJS，前后端 TypeScript 统一体验较好 |
| 数据库 | PostgreSQL | 适合关系数据、行程、好友、参与记录 |
| 缓存 | Redis | Token、验证码、热门列表、通知队列 |
| 推送 | 微信订阅消息 + App Push | 三端分别封装通知能力 |

### 4.3 三端差异原则

所有业务逻辑尽量复用，但平台能力必须隔离：

```txt
src/
  app.ts
  pages/
  components/
  modules/
  services/
  stores/
  utils/
  platform/
    index.ts
    weapp.ts
    rn.ts
```

平台差异统一通过 `platform` 层处理，不允许在业务页面中到处写平台判断。

例如：

- 微信登录、小程序订阅消息、分享给好友、生成小程序码：放在 `platform/weapp.ts`。
- App 推送、系统相册权限、定位权限、深链跳转：放在 `platform/rn.ts`。
- 页面业务只调用统一方法：`login() / shareTrip() / requestLocation() / subscribeTripReminder()`。

## 5. 产品核心模块

## 5.1 用户系统

### 目标

让每个用户拥有自己的身份、主页、朋友关系和行程空间。

### 功能

- 微信登录：小程序端 P0。
- 手机号登录：三端通用 P0。
- Apple 登录：iOS 上架前必须预留。
- 用户昵称、头像、简介、城市、兴趣标签。
- 用户主页：
  - 最近发布的行程。
  - 已参加的行程。
  - 好友数量。
  - 公开动态。
- 账号设置：
  - 修改资料。
  - 隐私设置。
  - 黑名单。
  - 注销账号。
  - 数据导出预留。

---

## 5.2 朋友关系系统

### 目标

不是做开放社交，而是做“熟人小圈子”。

### 功能

- 添加朋友：
  - 搜索 ID / 昵称。
  - 扫码添加。
  - 邀请链接添加。
  - 通过行程加入后申请成为好友。
- 好友申请：
  - 申请理由。
  - 同意 / 拒绝。
- 朋友分组：
  - 家人。
  - 同学。
  - 同事。
  - 旅行搭子。
  - 饭搭子。
  - 游戏搭子。
  - 自定义分组。
- 常约朋友：
  - 根据互动频率自动排序。
  - 用户可手动置顶。
- 黑名单 / 屏蔽 / 不让他看。
- 行程发布时选择可见范围：
  - 仅自己。
  - 指定好友。
  - 指定分组。
  - 好友可见。
  - 链接可见。

---

## 5.3 行程发布系统

### 目标

用户可以像发朋友圈一样发起一场活动，但内容是结构化的。

### 行程基础字段

| 字段 | 说明 |
|---|---|
| 标题 | 例如“周六青城山轻徒步” |
| 封面图 | 行程氛围图或地点图 |
| 行程类型 | 聚餐 / 旅行 / 运动 / 看展 / 游戏 / 学习 / 其他 |
| 开始时间 | 精确到日期和时间 |
| 结束时间 | 可选 |
| 地点 | 支持地图位置、线上链接或手动填写 |
| 人数限制 | 可选，如 4 人、8 人、不限 |
| 费用说明 | AA、我请客、免费、预估预算 |
| 可见范围 | 指定好友 / 分组 / 链接 |
| 参与规则 | 直接加入 / 需要确认 / 候补 |
| 行程描述 | 纯文本 / Markdown 子集 / 富文本预留 |
| 准备清单 | 要带什么、注意事项 |
| 标签 | 自驾、露营、拍照、轻松、低预算等 |

### 发布模式

- 快速发布：只填标题、时间、地点、描述。
- 详细发布：包含路线、预算、清单、日程安排。
- 模板发布：复用历史行程模板。
- 草稿模式：未完成前可保存。
- 预约发布：后续版本预留。

## 5.4 行程结构化编辑器

### 目标

让用户可以把一场复杂行程拆成清楚的步骤。

### 模块

- 行程时间线：
  - 09:00 集合。
  - 10:30 到达目的地。
  - 12:00 午餐。
  - 14:00 自由活动。
  - 17:00 返程。
- 地点列表：
  - 集合点。
  - 途经点。
  - 目的地。
  - 备选地点。
- 准备清单：
  - 防晒。
  - 充电宝。
  - 身份证。
  - 雨伞。
  - 相机。
- 预算拆分：
  - 交通。
  - 餐饮。
  - 门票。
  - 住宿。
  - 其他。
- 注意事项：
  - 天气。
  - 穿着。
  - 安全提醒。
  - 集合迟到规则。

## 5.5 参加与约定系统

### 目标

朋友看到行程后，可以明确表达自己的参与状态，而不是只在聊天里说“看情况”。

### 参与状态

| 状态 | 含义 |
|---|---|
| 想去 | 有兴趣，但未确认 |
| 确认参加 | 确定参与 |
| 暂不确定 | 需要看时间 |
| 候补 | 人满后排队 |
| 不参加 | 明确拒绝 |
| 已取消 | 原本参加后退出 |

### 功能

- 一键参加。
- 发起人审核参加申请。
- 人数上限。
- 候补队列。
- 参加人备注，例如“我可以开车”“我带相机”。
- 发起人修改参与状态。
- 退出活动。
- 参加截止时间。
- 活动变更通知。

---

## 5.6 行程互动系统

### 目标

让行程讨论聚焦在该行程内部，而不是散落在微信群聊天中。

### 功能

- 评论。
- 回复评论。
- 表情回应。
- 投票：
  - 选择日期。
  - 选择地点。
  - 选择餐厅。
  - 选择预算。
- 发起人置顶公告。
- 重要信息更新记录。
- 参与者提建议：
  - 推荐地点。
  - 推荐路线。
  - 推荐装备。
- 行程内简易聊天预留。

---

## 5.7 通知与提醒系统

### 目标

让用户不会错过邀请、变更和出发提醒。

### 三端通知策略

| 场景 | 微信小程序 | Android / iOS |
|---|---|---|
| 好友申请 | 服务通知 / 站内通知 | App Push / 站内通知 |
| 行程邀请 | 订阅消息 / 站内通知 | App Push / 站内通知 |
| 活动开始前提醒 | 订阅消息 | 系统推送 |
| 行程变更 | 订阅消息 / 站内通知 | App Push / 站内通知 |
| 评论回复 | 站内通知 | App Push / 站内通知 |

### 功能

- 行程开始前提醒：1 天前、3 小时前、30 分钟前。
- 参加状态变化提醒。
- 发起人修改时间 / 地点提醒。
- 新评论 / 回复提醒。
- 好友申请提醒。
- 系统公告。
- 通知设置：
  - 全部开启。
  - 只接收重要提醒。
  - 免打扰。

---

## 5.8 分享与邀请系统

### 目标

让行程可以自然地在微信和 App 内传播，但仍然保持权限边界。

### 微信小程序 P0 能力

- 分享到微信好友。
- 分享到微信群。
- 生成行程小程序码。
- 通过分享卡片进入行程详情。
- 未登录用户进入后引导登录。
- 链接可见行程支持申请加入。

### App 端能力

- 分享行程链接。
- 生成邀请海报。
- App 内好友邀请。
- Deep Link / Universal Link 预留。

### 权限控制

- 私密行程不可被二次转发查看。
- 链接行程可设置：
  - 任何获得链接的人可申请。
  - 仅好友可申请。
  - 仅指定人可查看。
- 被移除的参与者不可继续访问私密行程。

---

## 5.9 行程记录与回顾系统

### 目标

让完成后的行程沉淀为用户自己的生活记录。

### 功能

- 行程结束后自动进入“已完成”。
- 用户可上传照片。
- 参与者可共同补充回顾。
- 发起人可生成活动总结。
- 支持“再次发起类似行程”。
- 支持生成海报：
  - 标题。
  - 日期。
  - 参与者。
  - 精选照片。
  - 简短总结。
- 个人主页展示已完成行程。
- 年度行程回顾预留。

## 8. MVP 范围

## 8.1 MVP 必须完成

MVP 的目标不是做完整社交平台，而是验证“发布行程 → 分享给朋友 → 朋友参加 → 行程内确认”的闭环。

### P0 功能

- 微信小程序端首发。
- 微信登录 / 手机号登录。
- 用户资料。
- 创建行程。
- 行程详情。
- 行程列表。
- 分享行程到微信好友 / 群。
- 朋友通过分享进入行程。
- 申请参加 / 确认参加。
- 参与者列表。
- 评论。
- 行程修改通知。
- 我的行程。
- 基础权限：公开链接 / 好友可见 / 私密。

### MVP 暂不做

- 完整 App 推送。
- 复杂好友分组。
- 复杂投票。
- 预算多人分摊。
- AI 自动生成行程。
- 年度回顾。
- 多平台完整体验一致性。
- 桌面端。

- 行程圈首页。
- 创建行程。
- 行程详情。
- 参加状态。
- 评论。

## V0.2 微信小程序 MVP

目标：形成完整小程序闭环。

- 微信登录。
- 手机号绑定。
- 头像昵称设置。
- 参与者管理。
- 行程可见范围。
- 行程修改提醒。
- 邀请卡片。

## 13. 权限与隐私原则

这是熟人社交产品，隐私必须从一开始设计。

### 必须支持

- 行程可见范围控制。
- 被邀请人不能默认看到所有发起人历史行程。
- 私密行程不能被非授权用户查看。
- 好友关系需要双方确认。
- 用户可以删除自己的评论。
- 发起人可以移除不合适的参与者。
- 黑名单用户不能查看、评论、申请参加。
- 用户可以注销账号。

### 上线前必须检查

- 微信小程序用户隐私协议。
- App 隐私政策。
- 用户协议。
- 定位权限说明。
- 相册权限说明。
- 通知权限说明。
- iOS Apple 登录合规。

## 15. 代码 Agent 执行原则

当把本项目交给代码 Agent 执行时，必须遵守以下原则：

1. **先做微信小程序 MVP，不要一开始同时铺开三端。**
2. **所有业务代码必须 TypeScript 化。**
3. **页面层只处理展示与交互，不直接写复杂业务逻辑。**
4. **所有接口请求必须经过统一 request 封装。**
5. **所有平台差异必须进入 platform 层。**
6. **组件必须按业务模块拆分。**
7. **行程详情页是核心页面，必须优先打磨。**
8. **发布行程流程必须尽可能短。**
9. **不要过早引入复杂动画和复杂 UI。**
10. **不要过早做 AI、支付、复杂推荐。**
11. **先闭环，再美化，再扩展。**
12. **任何字段和接口都要考虑三端复用。**
13. **所有状态都要有空态、加载态、错误态。**
14. **所有用户生成内容都要预留审核与举报能力。**
15. **隐私权限不允许后补，必须从第一版开始设计。**

## 17. 最终验收标准

当项目达到终极 Goal 时，应满足以下标准：

- 用户可以在微信小程序、Android、iOS 三端登录同一个账号。
- 用户可以添加好友、管理好友、按好友或分组设置行程可见范围。
- 用户可以快速发布一场行程。
- 用户可以创建复杂行程，包括时间线、地点、清单、预算、注意事项。
- 用户可以把行程分享给微信好友、微信群或 App 内好友。
- 朋友可以查看行程并选择想去、确认参加、暂不确定、候补或不参加。
- 发起人可以审核、管理、移除参与者。
- 行程变更后参与者可以收到提醒。
- 行程内部可以评论、投票、协作准备。
- 活动结束后可以生成回顾和再次发起。
- 三端核心体验一致，平台差异被良好封装。
- 数据安全、隐私权限、举报审核、账号注销具备基本能力。
- 整个产品可以真实支撑用户组织朋友聚会、旅行、运动、看展等活动。

## 18. 项目最终定义

这个项目的最终形态不是“另一个朋友圈”，也不是“另一个日历”，而是：

> 一个以朋友关系为基础、以行程为内容载体、以参加约定为核心动作、以回顾沉淀为长期价值的轻社交协作产品。

第一阶段要极度克制，只做一件事：

> 让一个人可以方便地发起一场行程，让朋友可以清楚地决定要不要一起参加。

只要这个闭环成立，再逐步扩展好友系统、投票系统、清单系统、回顾系统和 AI 能力。
